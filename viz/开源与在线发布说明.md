# 开源与在线发布：规则、要求与实现方式

> 本文档记录「在 GitHub 上开源且所有人可直接在浏览器中查看可视化、无需本地部署」的规则、约束与具体实现方式，供维护者与贡献者遵循。

---

## 一、规则与目标

### 1.1 核心规则

- **开源载体**：项目托管于 GitHub，仓库公开，任何人可 clone / fork。
- **在线可见**：可视化页面（T-PA-G 等）须支持**在浏览器中直接打开、交互**，无需在本地起服务或执行构建命令。
- **零本地部署**：访客不应被要求执行 `npm install`、`npx serve`、`python -m http.server` 等；仅通过点击链接即可使用。

### 1.2 目标用户行为

| 角色     | 期望行为 |
|----------|----------|
| 访客     | 打开文档中的链接 → 直接看到可视化并可与节点交互（点击查看说明）。 |
| 贡献者   | 修改 viz 或文档后 push 到 GitHub → 在线页面随发布流程自动更新（若已配置自动发布）。 |
| 维护者   | 按本文档的「实现方式」完成一次性的 GitHub Pages（或等效）配置，之后仅需维护内容。 |

---

## 二、技术要求与约束

### 2.1 技术约束

- **仅静态资源**：当前可视化为纯前端（HTML + CSS + JavaScript），无服务端逻辑，符合 GitHub Pages 等静态托管的要求。
- **无构建强依赖**：不要求必须使用打包工具（Webpack / Vite 等）；若后续引入构建步骤，须保证构建产物可被静态托管且文档中说明。
- **资源同源**：脚本与数据（如 `data.js`）与 `index.html` 同目录或同站点下，避免因 `file://` 或跨域导致无法加载；通过 GitHub Pages 访问时均为同源。

### 2.2 不采用的方案及原因

| 方案                 | 不采用原因 |
|----------------------|------------|
| 仅提供仓库内 HTML 链接 | GitHub 仓库内直接打开 HTML 会展示源码，不执行页面，无法「直接可视化」。 |
| 要求访客本地起服务   | 与「零本地部署」规则冲突。 |
| 依赖需登录的第三方服务 | 增加使用门槛，不利于开源可及性。 |

### 2.3 可选扩展（不改变主规则）

- 若后续使用 **Netlify / Vercel / Cloudflare Pages** 等：可保留「从 GitHub 仓库自动部署」的流程，但**主推荐入口**仍建议优先使用 GitHub 生态内的 GitHub Pages，并在 README 中明确写出在线访问链接。
- 若使用自定义域名：在实现方式中增加「自定义域名」小节即可，规则与目标不变。

---

## 三、实现方式：GitHub Pages

### 3.1 适用条件

- 仓库为 GitHub 上的公开仓库。
- 可视化资源为静态文件，位于仓库内某一目录（如 `docs/rule/viz/` 或 `docs/` 下）。

### 3.2 目录与 URL 约定

- **当前结构**（示例）  
  - 可视化入口：`docs/rule/viz/index.html`  
  - 同目录：`data.js`、`viz1-flow.js` ~ `viz5-force.js`、`README.md` 等。
- **GitHub Pages 源**（二选一，推荐方案 A）  
  - **方案 A**：Source 选为 **main**（或默认分支）的 **/docs** 目录。  
    - 站点根路径对应仓库中的 `docs/`。  
    - 可视化访问路径：  
      `https://<用户名或组织>.github.io/<仓库名>/rule/viz/`  
      或  
      `https://<用户名或组织>.github.io/<仓库名>/rule/viz/index.html`
  - **方案 B**：Source 选为 **main** 的 **根目录**（/）。  
    - 站点根路径对应仓库根目录。  
    - 可视化访问路径：  
      `https://<用户名或组织>.github.io/<仓库名>/docs/rule/viz/`  
      或  
      `https://<用户名或组织>.github.io/<仓库名>/docs/rule/viz/index.html`

将上述「可视化访问路径」作为**唯一正式在线地址**写入 README 与本文档，避免歧义。

### 3.3 配置步骤（方案 A：/docs 作为站点根）

1. 打开仓库 **Settings → Pages**。
2. **Build and deployment**：  
   - **Source** 选 **Deploy from a branch**。  
   - **Branch** 选 **main**（或当前默认分支），**Folder** 选 **/docs**。  
   - 保存。
3. 等待 GitHub 完成首次部署（通常 1–2 分钟），顶部会显示站点 URL：  
   `https://<用户名或组织>.github.io/<仓库名>/`
4. 在 README 或本文档中写明：  
   - **在线可视化**：`https://<用户名或组织>.github.io/<仓库名>/rule/viz/`

### 3.4 配置步骤（方案 B：根目录作为站点根）

1. **Settings → Pages**。  
2. **Source**：**Deploy from a branch**；**Branch** 选 **main**，**Folder** 选 **/ (root)**。  
3. 在线根路径为仓库根目录；可视化地址为：  
   `https://<用户名或组织>.github.io/<仓库名>/docs/rule/viz/`

若仓库根目录下存在 `index.html`，则根 URL 会显示该页；否则可能为文件列表，需通过上述子路径访问 viz。

### 3.5 内容更新与发布

- 所有变更通过 **git push** 到对应分支即可；GitHub Pages 会自动从该分支的指定目录重新部署。
- 无需在本地执行构建或上传到其他平台即可更新在线页面（在未引入构建流程的前提下）。

## 四、文档与可发现性

### 4.1 必须写明的信息

- **主 README**（仓库根或 `docs/` 根）：  
  - 一句说明「T-PA-G 可视化可在线查看」；  
  - **在线链接**：上述 GitHub Pages 可视化 URL（如 `https://<owner>.github.io/<repo>/rule/viz/`）。
- **本目录 README**（`docs/rule/viz/README.md`）：  
  - 保留「如何打开」：优先写「在线访问」链接，其次写本地开发方式（如 `npx serve`、本地服务）。

### 4.2 可选

- 在「如何打开」中增加「若 GitHub Pages 未配置，可本地运行 `npx serve docs/rule/viz` 预览」的说明。
- 本文档（`开源与在线发布说明.md`）在 viz 目录下保留，便于后续维护者与贡献者查阅规则与实现方式。
